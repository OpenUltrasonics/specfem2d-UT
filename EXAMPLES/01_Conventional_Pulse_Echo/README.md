## Modifying the Simulation Setup

All simulation parameters are defined in the `parameter.yaml` file located in this example directory.

### Material Properties

```yaml
material:
  vp: 5950.0
  vs: 3200.0
  rho: 7850.0
```

| Parameter | Description | Units |
|-----------|-------------|-------|
| `vp` | P-wave velocity | m/s |
| `vs` | S-wave velocity | m/s |
| `rho` | Density | kg/m³ |

### Boundary Conditions

```yaml
boundaries:
  top: free
  bottom: absorbing
  left: absorbing
  right: absorbing
```

Each boundary can be set to either `free` or `absorbing`. In a typical pulse-echo setup, the top surface (where the transducer sits) is `free`, and all other boundaries are `absorbing` to suppress artificial reflections from the model edges.

### Transducer Parameters

```yaml
transducer:
  f0: 2250000.0
  aperture: 0.002
  center_x: 0.025
  zs: 0.050
  total_factor: 1.0e9
  apodization: "uniform"
```

| Parameter | Description |
|-----------|-------------|
| `f0` | Centre frequency of the transducer (Hz) |
| `aperture` | Physical aperture width of the transducer (m) |
| `center_x` | Horizontal position of the transducer centre (m) |
| `zs` | Vertical (depth) position of the source (m) |
| `total_factor` | Source amplitude scaling factor |
| `apodization` | Element weighting: `uniform` or `gaussian` |

### Source Wavelet

```yaml
use_custom_source: True
window_type: "hanning"
num_cycles: 3
```

| Parameter | Options | Description |
|-----------|---------|-------------|
| `use_custom_source` | `True` / `False` | `True` → custom tone burst; `False` → default Ricker wavelet |
| `window_type` | `hanning`, `gaussian`, `rectangular` | Envelope applied to the tone burst |
| `num_cycles` | Integer (typically 2–5) | Number of cycles in the tone burst. Fewer cycles give better axial resolution; more cycles give a narrower frequency bandwidth. |

### Simulation Controls

```yaml
simulation:
  total_time: 0.000020
  mesh_name: "steel_sdh"
```

| Parameter | Description |
|-----------|-------------|
| `total_time` | Total simulation recording time (seconds). Should be long enough to capture all echoes of interest. |
| `mesh_name` | Name of the mesh file to use (without extension). Must match the `.msh` file generated by Gmsh. |

After modifying `parameter.yaml`, rerun the example:

```bash
bash run_this_example.sh
```

## Modifying the Geometry

The specimen geometry is defined using a Gmsh `.geo` file located in this example directory.

To simulate a different geometry — for example, a different defect type, position, or specimen shape — modify the corresponding `.geo` file and regenerate the mesh before rerunning.

Geometry files can be created or edited using:

- **[Gmsh](https://gmsh.info/)** (recommended) — a free and open-source mesh generation tool with both a GUI and scripting interface. The GUI allows you to visualise the geometry interactively before meshing.
- **Any text editor** — `.geo` files are plain text and can be edited directly. The Gmsh scripting language is well documented at [gmsh.info/doc/texinfo/gmsh.html](https://gmsh.info/doc/texinfo/gmsh.html).
- **AI coding assistants or LLMs** — tools like Claude, ChatGPT, or GitHub Copilot can generate `.geo` files from a plain-English description of the desired geometry, which can then be refined manually.

After updating the geometry, regenerate the mesh and rerun:

```bash
bash run_this_example.sh
```
